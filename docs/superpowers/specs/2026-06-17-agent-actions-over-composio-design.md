# Agent Actions over Composio — Design Spec

**Date:** 2026-06-17
**Status:** Approved design, ready for implementation plan
**Related:** `docs/CHAT_FIRST_AGENTIC_INTERFACE_FEATURE_ROADMAP.md` (§4.11, §4.12, §9), the Approvals slice (Group 6), `src/lib/agent/approvals/`, `src/lib/composio/`

## 1. Context & goal

The Approvals slice gave the agent a human-in-the-loop gate: it drafts an external action, persists a *pending* `approval_requests` row, and the real side-effect runs only after the user approves. This slice makes that real for the product's primary channel — **the user's connected Gmail** — and lays the action-layer foundation so GitHub/Slack/Drive actions become cheap follow-ons.

Goal: a user connects Gmail **once**, asks the agent to email someone, sees a confirm-before-send card (the generative-UI popup), approves, and the mail goes out from their own account — with the agent only ever offering actions the user has actually connected.

## 2. Decisions (settled in brainstorming)

1. **Substrate = Composio, connect-once.** All agent reads *and* writes go through Composio, keyed by the app `userId`. Connecting Gmail once serves both ingestion and sending. The agent shell's Connect buttons move from native Google OAuth (`/api/accounts/connect`) to Composio (`/api/composio/connect`).
2. **Scope = foundation + Gmail send, end-to-end.** Build the action-tool framework + connection awareness + Connect-UX repoint, and make `gmail_send_email` work start to finish. GitHub/Slack/Drive actions are deliberately out of this slice.
3. **Approval authority = requester-only.** Only the user whose run drafted an action may approve it; it runs from their connection. Enforced as `approval.requested_by === approver` (plus the existing membership check). Delegation/shared approval deferred.
4. **Action layer = registry-driven *named* tools + connection-gating.** The executor registry stays the single source of truth, but each entry is exposed to the model as its **own named, typed Mastra tool** (`gmail_send_email(to, subject, body)`), auto-generated by a factory — not as one generic `request_approval(actionType, payload)`. Only tools for *connected* toolkits are attached per run.

### 2.1 Why named tools (the legibility resolution)

A single generic tool is less legible as actions grow: the model must pick the right `actionType` string and shape an opaque `payload`. Generating one named tool per registry entry **eliminates** the wrong-actionType failure mode (the tool name *is* the action) and the malformed-payload mode (the model sees the real typed schema), with no per-action boilerplate (one `makeActionTool(entry)` factory). Remaining risks are handled structurally, not deferred:

- **Under/over-triggering** (claiming it sent; acting on read-only) — an instructions concern, unaffected by tool shape; handled by the standing "never claim an external action — draft and call the tool" rule + examples.
- **Tool-list bloat** at dozens of actions — handled by **connection-gating** (only connected toolkits' tools exist per run), consistent `toolkit_action` naming, and an **optional toolkit-router tier deferred** until the catalog is large.
- **Floor:** server validates actionType + payload at request *and* execute time; nothing runs without approval. A model mistake is at worst a visible failed call, never a wrong action firing.

## 3. Architecture & trust path

```
Connect once (Composio)
   Connect button ─▶ /api/composio/connect?toolkit=gmail ─▶ consent ─▶ callback
        ├─ powers reads  (GMAIL_FETCH_EMAILS ingestion, existing)
        └─ powers writes (GMAIL_SEND_EMAIL, this slice)        ← one connection

Per run (/api/agent)
   connected = listConnectedToolkits(userId)            // concurrent w/ setup, ~60s cache
   tools = { ...baseTools, ...buildActionTools(connected) }   // only connected toolkits
   instructions += "Action tools available now: Gmail → gmail_send_email. If asked for
                    an action not listed, tell the user to connect it in Settings → Apps."

Draft → approve → act
   user asks ─▶ agent calls gmail_send_email(to,subject,body)
        └─▶ validate → createApprovalRequest(pending) → approval card renders in chat
   user clicks Approve ─▶ POST /api/agent/approvals/[id]
        ├─ AUTHZ: approver === requested_by (NEW) + membership
        ├─ atomic claim pending→approved (built)
        └─ executor runs GMAIL_SEND_EMAIL via Composio from requested_by's connection
        └─▶ status: executed ─▶ card shows "Approved · email sent"
```

## 4. Components

**Action layer (server)**
- `src/lib/agent/approvals/executors.ts` *(mod)* — registry is the source of truth. Each entry gains a `description`; tool name derived as `${toolkit}_${actionType}` → `gmail_send_email`. Add `availableActions(connectedToolkits)`.
- `src/mastra/tools/action-tools.ts` *(new, replaces `approval-tools.ts`)* — `makeActionTool(entry)` builds a Mastra tool whose `inputSchema` is the entry's typed schema and whose `execute` does validate → `createApprovalRequest` → return the approval view. `buildActionTools(connected)` returns the tool set filtered to connected toolkits.

**Connection awareness (server)**
- `src/app/api/agent/route.ts` *(mod)* — fetch connected toolkits (concurrent, cached), attach connection-gated action tools, inject the available-actions instruction line.
- `src/app/api/agent/connections/route.ts` *(new)* — `GET` → the user's connected toolkits, for the client.

**Trust boundary (server)**
- `src/app/api/agent/approvals/[id]/route.ts` *(mod)* — add requester-only check (`requested_by === user.id`, else 403).

**Connect-once UX + @mentions (client)**
- `src/components/agent-shell/agent-plugins-view.tsx` *(mod)* — `onConnect` → `/api/composio/connect?toolkit=<key>`; Connected state from the connections route.
- `src/components/agent-shell/agent-mention-menu.tsx` *(mod)* — filter integration mentions to connected toolkits; a "Connect more apps →" row links to the plugins view.

**Reused unchanged:** approval `store.ts`, the approval card, `/dev/approvals-preview`, `AGENT_APPROVALS_DRY_RUN`.

## 5. Connection-awareness details

- **Connect once:** Composio connections are keyed by the app `userId` (the id ingestion already uses), so one Gmail connection serves reads + writes.
- **Per-run gating:** `listConnectedToolkits(userId)` is best-effort (`→ []` on error), run concurrently with the existing membership/persona queries, with a short per-user cache (~60s) to avoid taxing every turn.
- **Not-connected path, defended at four layers:** (1) agent declines and points to Settings (from injected instructions); (2) `@gmail` not offered in the mention menu; (3) plugins view shows Connect; (4) executor re-checks connection at execute time as a backstop.

## 6. Trust boundaries

1. Model can't act directly — tools only create *pending* approvals; nothing executes in the stream.
2. Requester-only approval (`requested_by === approver`), plus membership.
3. Identity from server session + RequestContext, never model args; action runs from `requested_by`'s connection.
4. Payload loaded from the DB on approve — the client sends only the decision.
5. Atomic pending→approved claim guards against double-execute.
6. Validate actionType + payload at request *and* execute time; unknown actions rejected.
7. Connection re-checked at execute (backstop).
8. Mentions are hints, not permissions (§9.1) — menu gating is UX; the real gate is the executor + requester/membership checks.

## 7. Testing strategy

| Tier | Needs | Proves | Affordance |
|------|-------|--------|------------|
| A — now | nothing | card states; registry generates one tool per connected toolkit; unknown action rejected; tsc/lint | `/dev/approvals-preview` + registry check |
| B — DB up, no Gmail | dev flags | full loop (draft→card→approve→executed) with nothing sent; connection-awareness | `AGENT_APPROVALS_DRY_RUN=true` + new `AGENT_FAKE_CONNECTED=gmail` |
| C — real (acceptance) | one Gmail connected | real end-to-end send; **verifies `GMAIL_SEND_EMAIL` arg shape** | live Composio connect |

**`AGENT_FAKE_CONNECTED`** (dev-only, non-production) forces the connected-toolkit list so the full loop + connection-awareness is testable without a real connection — leaving only the Composio arg-shape as a genuine connect-required check.

**Acceptance:** connect Gmail once (shows Connected) → "email the growth team the launch update" → agent drafts → card → Approve → mail arrives at a test address, card reads "Approved · email sent" → disconnected case points to Settings and hides `@gmail` → a second user cannot approve the action.

## 8. Scope / non-goals

- **In:** action-tool framework, connection awareness (agent + @mentions), Connect-UX → Composio, `gmail_send_email` end-to-end, requester-only approval, dev test flags.
- **Out:** GitHub/Slack/Drive actions (cheap follow-ons); the toolkit-router tier; collapsing the legacy native-OAuth system (`external_accounts`, campaign-email `/api/emails`) — only the agent shell's Connect buttons move to Composio this slice; delegated/role-gated approval.

## 9. Open unknowns (resolve in planning/implementation)

1. **Per-run tool gating mechanism** — whether Mastra's `Agent` accepts a per-request tool set / dynamic tools keyed on RequestContext (preferred), else build the tool set in the route / reconstruct the agent per request. Affects only how `buildActionTools(connected)` is attached.
2. **Composio `GMAIL_SEND_EMAIL` arg shape** (`recipient_email` / `extra_recipients` / `is_html`) — unverified; confirm against a live connection (Tier C).
3. **Branch overlap** — the paused `feat/agentic-retrieval-loop` also edits `supervisor-agent.ts`; sequence/rebase to avoid collision.

## 10. Changes to already-built (Approvals) code

- `approval-tools.ts` (single generic `requestApproval`) → `action-tools.ts` (registry-driven `makeActionTool` factory + `buildActionTools`).
- `send_email` executor already targets Composio `GMAIL_SEND_EMAIL` + dry-run + connection check — kept; arg shape still to verify.
- Approval API gains the requester-only check.
- `supervisor-agent.ts` stops importing the generic tool; the route attaches connection-gated action tools per run.
